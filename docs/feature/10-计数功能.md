统计文章浏览数（PV, Page View）看似简单（每次请求数据库 +1），但在实际生产环境中，为了兼顾**准确性**、**高性能**和**防作弊**，需要采取更合理的方案。

以下是一个从基础到进阶的统计方案建议：

### 1. 核心架构：缓存异步更新
直接在用户访问时操作数据库（如 `UPDATE articles SET views = views + 1`）在高并发下会产生数据库压力，并导致行锁竞争。

*   **方案**：使用 **Redis** 作为缓冲。
*   **流程**：
    1.  用户访问文章，后端在 Redis 中执行 `INCR article:views:{id}`。
    2.  通过定时任务（如每 5-10 分钟）或消息队列，将 Redis 中的增量数据同步回 MySQL 数据库进行持久化。

---

### 2. 统计口径：如何定义“一次浏览”？
为了防止刷量，不能简单地刷新页面就加 1。通常有以下几种去重策略：

*   **IP 去重**：同一 IP 在 24 小时内访问多次只计一次。
    *   *缺点*：同一局域网下的用户 IP 相同；动态 IP 会变。
*   **用户 ID 去重**：针对登录用户，按 `user_id` 统计。
    *   *优点*：最准确。
*   **Cookie/设备指纹去重**：针对游客，在浏览器写入一个唯一标识（UUID），有效期 24 小时。
    *   *实现*：在 Redis 中记录 `view_log:{article_id}:{visitor_id}`，并设置过期时间。如果该 Key 已存在，则不增加计数。

---

### 3. 防作弊与过滤（反作弊）
为了保证数据的含金量，需要剔除无效流量：

*   **过滤爬虫**：通过检查 HTTP 请求的 `User-Agent`，过滤掉百度、谷歌等搜索引擎的爬虫。
*   **停留时长限制**：通过前端埋点，只有当用户页面停留超过 5 秒或滚动超过一定比例时，才发送增加计数的请求。
*   **频率限制**：同一用户/IP 在短时间内（如 1 秒内）频繁请求，视为作弊。

---

### 4. 进阶方案：大数据量下的处理
如果你的网站访问量极大，Redis 的 Key 数量过多会占用大量内存，可以考虑：

*   **HyperLogLog (HLL)**：
    *   Redis 的 HyperLogLog 是一种概率数据结构，用于基数统计。
    *   *优点*：极省内存（每个 Key 仅占 12KB），适合统计 UV（独立访客）。
    *   *缺点*：有 0.81% 的标准误差，不适合要求 100% 精确的场景。
*   **布隆过滤器 (Bloom Filter)**：
    *   用于快速判断一个用户是否已经访问过该文章，从而决定是否计入统计。

---

### 5. 推荐的组合方案

针对大多数中大型应用，建议采用以下流程：

1.  **前端埋点**：页面加载完成后，异步调用统计接口。
2.  **后端校验**：
    *   获取用户唯一标识（登录用 `uid`，未登录用 `Cookie` + `IP`）。
    *   检查 Redis 缓存：`EXISTS view_check:{article_id}:{visitor_id}`。
3.  **计数增加**：
    *   如果不存在，则 `SETEX view_check:... 86400 "1"`（标记今天已读）。
    *   执行 `INCR article:views:{id}`。
4.  **异步持久化**：
    *   使用 `Celery` 或 `Spring Task` 定时将 Redis 计数刷入数据库。
5.  **展示优化**：
    *   读取文章时，直接从 Redis 获取当前计数，并与数据库基础值相加，保证用户看到的是实时数据。

### 总结
*   **简单场景**：Redis `INCR` + 定时同步数据库。
*   **严谨场景**：用户标识校验 + 爬虫过滤 + 异步队列。
*   **海量 UV 场景**：Redis HyperLogLog。
