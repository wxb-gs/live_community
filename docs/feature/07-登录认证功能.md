# 06 — 登录认证功能

## 设计模式

登录模块使用了以下设计模式，每种模式承担明确的职责：

### 1. 策略模式（Strategy Pattern）— 核心分发

**问题**：系统需要支持密码、微信、淘宝、短信四种登录方式，每种方式的认证逻辑完全不同（查数据库 vs 调 OAuth API vs 验短信码），但调用方（Controller）不应关心这些细节。

**方案**：定义 `LoginStrategy` 接口，每种登录方式一个实现类，由 `LoginStrategyFactory` 根据请求中的 `type` 字段分发到对应策略。

```
LoginStrategy (接口)
  └─ authenticate(LoginRequest) → UserEntity

AbstractLoginStrategy (抽象类, 模板方法)
  ├─ PasswordLoginStrategy    — 用户名 + BCrypt 密码验证
  ├─ WechatLoginStrategy      — 微信 OAuth code 换 token + 用户信息
  ├─ TaobaoLoginStrategy      — 淘宝 OAuth code 换 token + 用户信息
  └─ PhoneSmsLoginStrategy    — 短信验证码验证 + 自动注册

LoginStrategyFactory
  └─ getStrategy(LoginType) → LoginStrategy
```

**关键代码**：`LoginStrategyFactory` 在构造时注入所有 `LoginStrategy` bean，构建 `Map<LoginType, LoginStrategy>` 映射表，O(1) 查找。

### 2. 模板方法模式（Template Method Pattern）— 公共逻辑复用

**问题**：四个策略都需要相同的日志记录和空值检查逻辑，直接写在每个策略里会重复。

**方案**：`AbstractLoginStrategy` 提供 `authenticate()` 骨架方法，子类只需实现 `doAuthenticate()`。

```
authenticate(req)  ← 模板方法（final）
  ├─ log.info("登录请求...")
  ├─ user = doAuthenticate(req)  ← 子类实现
  ├─ if (user == null) throw BadCredentialsException
  └─ return user
```

### 3. 工厂模式（Factory Pattern）— 策略创建与管理

`LoginStrategyFactory` 将策略的创建、注册、查找集中管理。Spring 的依赖注入自动收集所有 `LoginStrategy` 实现，工厂仅负责索引和分发。

### 4. 建造者模式（Builder Pattern）— JWT 构造

JJWT 库使用 Fluent Builder API 构造和解析 JWT：

```java
// 签发
Jwts.builder()
    .claims(payload)
    .subject(userId)
    .issuedAt(now)
    .expiration(expiry)
    .signWith(secretKey)
    .compact();

// 验证
Jwts.parser()
    .verifyWith(secretKey)
    .build()
    .parseSignedClaims(token);
```

### 5. 责任链模式（Chain of Responsibility）— 请求过滤

Spring Security 的 `SecurityFilterChain` 本质是责任链。`JwtAuthFilter`（继承 `OncePerRequestFilter`）被插入到 `UsernamePasswordAuthenticationFilter` 之前，按顺序执行：

```
请求 → JwtAuthFilter → UsernamePasswordAuthenticationFilter → ... → Controller
```

白名单路径 `/api/auth/**` 在 `JwtAuthFilter` 内部跳过，其他路径必须携带有效 JWT。

### 6. Provider 模式 — 多种认证令牌支持

Spring Security 的 `ProviderManager` 管理多个 `AuthenticationProvider`，每个 Provider 声明自己支持的令牌类型：

| Provider | 支持的 Token 类型 |
|----------|-------------------|
| `DaoAuthenticationProvider` | `UsernamePasswordAuthenticationToken` |
| `WechatAuthenticationProvider` | `WechatCodeAuthenticationToken` |
| `TaobaoAuthenticationProvider` | `TaobaoCodeAuthenticationToken` |
| `PhoneSmsAuthenticationProvider` | `PhoneSmsAuthenticationToken` |

### 7. 门面模式（Facade Pattern）— 控制器简化

`AuthController` 作为认证子系统的门面，对外暴露统一的 REST API（`/api/auth/login`、`/api/auth/register`、`/api/auth/refresh`、`/api/auth/logout`、`/api/auth/me`），内部委托给 `TokenService`、`LoginStrategyFactory`、`SmsService` 等多个服务，调用方只需与 Controller 交互。

### 8. DTO 模式（Data Transfer Object）— 传输层解耦

请求和响应使用专用 DTO（`LoginRequest`、`LoginResponse`、`RegisterRequest`、`RefreshRequest` 等），与数据库 Entity 分离，避免内部数据结构泄露到 API 层。

## 核心流程

### 密码登录

```
1. 前端 POST /api/auth/login  { type:"password", username, password }
2. gateway JwtAuthFilter 识别 /api/auth/** 白名单，放行
3. AuthProxyController 转发到 auth-service
4. AuthController.login() → LoginStrategyFactory.getStrategy("password")
5. PasswordLoginStrategy.doAuthenticate():
   a. UserRepository.findByUsername(username)
   b. BCryptPasswordEncoder.matches(password, user.password)
   c. 检查 status != DISABLED
6. TokenService.issueTokens(user):
   a. issueAccessToken() → JWT (HMAC-SHA256, 15min TTL, claims: sub/username/nickname/avatar)
   b. issueRefreshToken() → UUID 存入 Redis (key: refresh:<uuid>, 7天 TTL)
7. 返回 LoginResponse { accessToken, refreshToken, expiresIn, user }
8. 前端 saveTokens() → localStorage, 导航到主页
```

### 密码注册

```
1. 前端 POST /api/auth/register  { username, password, nickname }
2. AuthController.register():
   a. 检查用户名唯一性
   b. LeafRpcService.generateSegmentId("user") → 分布式 ID
   c. BCrypt 哈希密码
   d. UserRepository.save(user)
3. 自动调用 this.login() 完成登录（复用登录流程返回 token）
```

### 微信 OAuth 登录

```
1. 前端 GET /api/auth/wechat/url?redirectUri=... → 返回微信授权 URL
2. 浏览器重定向到微信授权页 → 用户扫码授权
3. 微信回调 redirectUri?code=xxx
4. WechatCallbackPage 提取 code, POST /api/auth/wechat/login
   ⚠ 注意: 此处前端调用 /api/auth/wechat/login, 但后端实际只有统一 /api/auth/login (type=wechat)
5. WechatLoginStrategy:
   a. code → 微信 API → access_token + openId
   b. access_token + openId → 微信 API → 用户信息
   c. UserWechatRepository 查找是否已绑定
   d. 不存在 → 自动注册新用户
   e. 返回 UserEntity
```

### 短信验证码登录

```
1. 前端 POST /api/auth/phone/send-code  { phone }
   → SmsService.sendCode(): 生成6位数字码, 存入 Redis (key: sms_code:<phone>, TTL 5min)
   → 当前为 mock 模式, 验证码仅记录日志
2. 用户输入验证码, POST /api/auth/login  { type:"phone", phone, smsCode }
3. PhoneSmsLoginStrategy:
   a. 从 Redis 验证 sms_code:<phone> 是否匹配
   b. UserPhoneRepository 查找 → 已存在直接返回, 不存在自动注册
```

### Token 刷新

```
1. 前端 POST /api/auth/refresh  { refreshToken }
2. AuthController.refresh():
   a. TokenService.validateAndGetUserId(refreshToken) → Redis 查询
   b. TokenService.revokeRefreshToken(refreshToken) → 撤销旧 token
   c. 根据 userId 重新签发 accessToken + refreshToken
3. ⚠ 前端 request() 拦截器遇到 401 直接跳转登录页, 未实现自动静默刷新
```

### 登出

```
1. 前端 POST /api/auth/logout  { refreshToken }
2. AuthController.logout():
   a. TokenService.revokeRefreshToken(refreshToken) → 删除 Redis 中的 refresh token
3. 前端 clearTokens() → 清除 localStorage, 跳转 /login
```

## JWT 验证链路

### 签发端（auth-service）

`TokenService` 使用 JJWT + HMAC-SHA256：

| Claim | 来源 |
|-------|------|
| `sub` | `user.id.toString()` |
| `username` | `user.username` |
| `nickname` | `user.nickname` |
| `avatar` | `user.avatar` |
| `iat` | 当前时间 |
| `exp` | `iat + access-token-ttl`（默认 900s） |

密钥来自 `application.yml` 的 `jwt.secret`，经 Base64 编码后作为 `SecretKeySpec`。

### 验证端（gateway）

`JwtAuthFilter` 在每个请求中：

1. 检查路径是否以 `/api/auth/` 开头 → 是则跳过
2. 从 `Authorization` 头提取 `Bearer <token>`
3. 用相同密钥验证签名
4. 提取 `sub` 和 `username` claims
5. 构造 `JwtAuthenticationToken` 并设置到 `SecurityContextHolder`
6. Controller 通过 `@AuthenticationPrincipal` 获取当前用户 ID

### 前端存储与携带

- `tokenStore.ts`：token 存储在 `localStorage`（键 `auth_tokens`），包含 `accessToken`、`refreshToken`、`expiresAt`
- `api/index.ts` 的 `request()` 函数：每次请求自动从 tokenStore 读取 access token，附加 `Authorization: Bearer <token>` 头
- `useCurrentUser.ts`：不调后端，直接从 JWT payload Base64 解码获取 `userId`、`username`、`avatar`
- `AuthGuard`（`App.tsx`）：包裹受保护路由，未认证时重定向到 `/login`

## 关键设计决策

### 为什么 Redis 存 refresh token 而非 JWT？

Refresh token 需要可撤销（登出时立刻失效）。JWT 是无状态的，签发后无法主动失效。用 Redis 存储 UUID → userId 映射，删除 key 即撤销，简单可靠。

### 为什么 gateway 和 auth-service 都验证 JWT？

gateway 是外部流量的唯一入口，必须验证。auth-service 作为内部服务，正常情况下只接收 gateway 转发，其 JwtAuthFilter 是纵深防御（defense in depth）。但当前 auth-service 的 `AuthSecurityConfig` 配置了 `.anyRequest().permitAll()`，实际访问控制完全由自定义 Filter 处理，未使用 Spring Security 的 URL 匹配能力。

### 为什么四种登录方式不使用独立的 REST 端点？

统一入口 `/api/auth/login` + `type` 字段，新增登录方式只需添加策略类和枚举值，Controller 代码零改动。这符合开闭原则（OCP）。

### 为什么用户 ID 使用 Leaf 分布式 ID 而非自增？

微服务架构下需要全局唯一 ID。Meituan Leaf 的 segment 模式（号段预分配）在保证唯一性的同时，相比每次访问数据库生成 ID 性能更高。

## 安全要点

| 项 | 措施 |
|----|------|
| 密码存储 | BCrypt 哈希，永不明文 |
| 传输安全 | JWT 签名验证（HMAC-SHA256），防篡改 |
| Token 有效期 | Access token 15min 短时效，Refresh token 7 天 |
| 会话管理 | 无状态 JWT + Redis refresh token，可横向扩展 |
| CSRF | 禁用（STATELESS 会话，无 Cookie） |
| 密码长度 | 6-100 字符限制 |

## 已知问题

1. **微信登录前后端路由不匹配**：前端 `wechatLogin()` 请求 `/api/auth/wechat/login`，但后端不存在此端点，应改为统一 `/api/auth/login`（`type: "wechat"`）
2. **无自动 token 刷新**：前端遇到 401 直接跳转登录页，未使用 refresh token 静默续期
3. **SMS 为 mock 模式**：`sms.provider: mock`，验证码仅日志输出，未接入真实短信服务
4. **JWT 密钥明文**：`jwt.secret` 直接写在 `application.yml` 中，生产环境应使用环境变量或密钥管理服务
