# 搜索功能 — Canal + Kafka + ES 实时索引

## 1. 功能概述

### 1.1 搜索系统架构

基于 **Canal + Kafka + Elasticsearch** 的实时索引同步链路，实现笔记和用户的全文搜索。笔记搜索使用 **Function Score** 融合 BM25 文本相关性评分与点赞/浏览业务权重，用户搜索使用纯 BM25。前缀联想通过 **Completion Suggester** 实现毫秒级搜索补全。

```
note-svc ──write──→ MySQL ──binlog──→ Canal ──publish──→ Kafka ──consume──→ search-sync-service ──index──→ ES
auth-svc ──write──→ MySQL ──binlog──┘                                                    │
                                                                                        │ query
                                                                                        ▼
gateway ←── search-service ←── ES (notes / users 索引)
   │
   ▼
front (React SearchPage)
```

### 1.2 浏览计数链路（高频写优化）

```
gateway (GET /api/note/detail) ──INCR──→ Redis (note:view:<id>)
                                              │
                                              │ 每 5min 定时任务 (ViewCountSyncScheduler)
                                              ▼
note-service ──UPDATE──→ MySQL ──binlog──→ Canal → Kafka → ES
```

直接写 MySQL 代价太大（每次点开详情都 +1），用 Redis 作缓冲计数器，5 分钟批量刷新到 MySQL，Canal 感知变更后同步到 ES。

### 1.3 三个索引 + 两种打分

| 索引 | 数据源 | 打分策略 | Suggest 字段 |
|------|--------|---------|-------------|
| `notes` | MySQL `note` 表 | Function Score = BM25 × log(1 + view×0.01 + like×0.5 + 2) | `title.suggest` |
| `users` | MySQL `user_info` 表 | 纯 BM25，`multi_match` on username + nickname^2 | `username.suggest`, `nickname.suggest` |

### 1.4 为什么两个索引分开？

笔记和用户是完全不同的实体——字段、打分逻辑、排序维度都不相同。分索引意味着：
1. 笔记搜索时不需要过滤 `_type`，查询更快
2. 各自的 mapping 和 analyzer 独立配置
3. 前端 tab 切换时只需改索引名，不调整查询逻辑
4. 后续可以独立扩展（如笔记加 `tags` 聚合、用户加 `bio` 搜索）

## 2. 详细实现分析

### 2.1 数据同步链路

**Canal → Kafka → ES 三步走**：

1. **Canal** 伪装成 MySQL slave，接收 binlog 事件，解析 INSERT/UPDATE/DELETE，将变更数据推送到 Kafka topic `search_sync`
2. **search-sync-service** 消费 Kafka 消息，`SearchSyncConsumer` 按表名分发：`note` 表 → `notes` 索引，`user_info` 表 → `users` 索引
3. **EsIndexService** 使用 `BulkProcessor` 批量写入 ES，500 条或 5 秒刷一次

```java
// SearchSyncConsumer.java — 消息分发
@KafkaListener(topics = "search_sync")
public void onMessage(String message) {
    CanalMessage msg = CanalMessage.fromJson(message);
    switch (msg.getTable()) {
        case "note":
            if ("DELETE".equals(msg.getType())) {
                esIndexService.deleteNote(msg.getData());
            } else {
                esIndexService.indexNote(msg.getData());
            }
            break;
        case "user_info":
            if ("DELETE".equals(msg.getType())) {
                esIndexService.deleteUser(msg.getData());
            } else {
                esIndexService.indexUser(msg.getData());
            }
            break;
    }
}
```

### 2.2 Function Score 打分公式

```
final_score = _score * log2(1 + view_count × 0.01 + like_count × 0.5 + 2)
```

**为什么这样设计？**

- `_score`：BM25 文本相关性（multi_match，title^3 + content + summary^2），保证搜索相关是主排序因子
- `view_count × 0.01`：每 100 次浏览 ≈ 1 倍 boost。浏览是弱信号——很多人点开看一段就关了
- `like_count × 0.5`：每 2 个赞 ≈ 1 倍 boost。点赞是强信号——用户明确表示有价值
- `+ 2`：基线偏置。0 浏览 0 点赞的新笔记，至少还有 `log(3) ≈ 1.58` 的分数，不会被压到 0
- `log2()`：对数平滑。防止热门笔记（万级浏览）完全碾压新笔记

**与纯 BM25 的对比**：

```
纯 BM25:
  "Spring Boot" 搜索 → 10 篇相关文章按文本相关性排序
  问题：一篇写得很差的旧文章可能排在高浏览量好文章前面

Function Score:
  BM25 保证相关性，业务权重在相关性基础上微调排序
  结果：相关性差不多的情况下，高质量内容排在前面
```

### 2.3 Completion Suggester 前缀联想

```java
// SuggestService.java
SearchResponse<Map> response = esClient.search(s -> s
    .index("notes", "users")
    .suggest(sug -> sug
        .suggesters("note_suggest", ss -> ss
            .prefix(q)
            .completion(comp -> comp
                .field("title.suggest")
                .size(5)
                .skipDuplicates(true)))
        .suggesters("user_suggest", ss -> ss
            .prefix(q)
            .completion(comp -> comp
                .field("username.suggest")
                .size(3)
                .skipDuplicates(true)))),
    Map.class);
```

**为什么用 Completion Suggester 而不是 edge n-gram？**

- Completion Suggester 是 FST（Finite State Transducer）数据结构，内存占用极低
- 查询 O(len) 时间复杂度，毫秒级响应
- 不支持模糊匹配（那是 Edge N-gram 的强项），但搜索补全场景不需要

**前缀联想的限制**：
- 只匹配前缀（"spring" → "Spring Boot"，但 "boot" 不会匹配 "Spring Boot"）
- 需要索引时指定 `completion` 类型字段
- 默认不返回 `_source`，需要 `_source: true` 才能拿到关联数据

### 2.4 浏览计数：Redis 原子增量 + 定时刷新

```java
// 写：gateway NoteController.detail()
redisTemplate.opsForValue().increment("note:view:" + noteId);

// 读+刷：note-service ViewCountSyncScheduler
@Scheduled(fixedRate = 300_000) // 5min
public void syncViewCounts() {
    Set<String> keys = redisTemplate.keys(VIEW_KEY_PREFIX + "*");
    for (String key : keys) {
        String countStr = redisTemplate.opsForValue().getAndDelete(key);
        if (countStr != null) {
            noteMysqlRepository.addViewCount(noteId, Integer.parseInt(countStr));
        }
    }
}
```

**关键设计决策**：
- `INCR` 是原子操作，无竞争条件
- `getAndDelete` 原子取走+删除，防止重复累加
- 异常被 catch 不中断 batch，单个 key 失败不影响其他
- 5 分钟间隔意味着最坏情况丢 5 分钟浏览数据

### 2.5 笔记 MySQL 双写

```java
// NoteService.java — createDraft / publishNote
noteRepository.save(entity); // Cassandra（评论文本）
noteMysqlRepository.upsert(  // MySQL（搜索元数据）
    entity.getId(), entity.getUserId(), entity.getTitle(), entity.getContent(),
    entity.getSummary(), null, null,
    entity.getStatus(), entity.getCreatedAt(), entity.getUpdatedAt()
);
```

`upsert()` 使用 `INSERT ... ON DUPLICATE KEY UPDATE`，幂等设计——重复调用不会出错。

## 3. ES Java Client 8.x API

本项目使用 `co.elastic.clients:elasticsearch-java:8.12.0`（新版 Java Client），所有查询用 Fluent DSL Builder：

```java
esClient.search(s -> s
    .index("notes")
    .query(q -> q
        .functionScore(fs -> fs
            .query(inner -> inner.bool(b -> b
                .must(m -> m.multiMatch(mm -> mm
                    .fields("title^3", "content", "summary^2")
                    .query(keyword)))
                .filter(f -> f.term(t -> t.field("status").value("PUBLISHED")))))
            .functions(fn -> fn.scriptScore(ss -> ss
                .script(sc -> sc.inline(in -> in
                    .source("_score * Math.log(...)"))))))),
    Map.class);
```

**旧版 vs 新版**：
- 旧版：`org.elasticsearch.client:elasticsearch-rest-high-level-client`（已弃用）
- 新版：`co.elastic.clients:elasticsearch-java`（Fluent API，类型安全）
- 新版所有 builder 方法都是 lambda 风格，IDE 有代码提示

## 4. 新模块架构

| 模块 | 端口 | Dubbo | 职责 |
|------|------|-------|------|
| `search-sync-service` | 8085 | 20885 (register=false) | 消费 Kafka，索引到 ES |
| `search-service` | 8086 | 20886 | 搜索/联想 API，Dubbo Provider |

**search-sync-service** 注册到 Nacos 但不注册 Dubbo（`register: false`），因为它只是消费者，不对外提供服务。

**search-service** 是 Dubbo Provider，被 gateway 通过 `@DubboReference SearchRpcService` 调用。

## 7. 设计模式

| 模式 | 应用 |
|------|------|
| **策略模式** | SearchPage tab 切换本质上切换了两个搜索策略（notes vs users），两种搜索服务使用不同的 Service 方法 |
| **适配器模式** | `EsDocMapper` 将 MySQL 行数据（JsonNode）适配为 ES 文档（Map） |
| **观察者模式** | Kafka pub/sub：Canal 发布变更事件，search-sync-service 订阅并处理 |
| **管道-过滤器** | Canal → Kafka → search-sync-service → ES 是一条数据处理管道 |
| **门面模式** | `SearchController` 对外暴露统一搜索接口，内部委托给 NoteSearchService / UserSearchService / SuggestService |
| **模板方法** | 所有 ES 查询服务都使用相同的 `try/catch → log.error → return empty` 错误处理模板 |

## 8. 面试准备

**Q: Canal 同步延迟怎么处理？**

> Canal 是 binlog 级别的实时同步，延迟通常在毫秒级（MySQL → Canal → Kafka → ES 的总延迟一般 < 1s）。如果出现延迟：
> 1. 检查 Kafka consumer lag（`kafka-consumer-groups --describe`）
> 2. Canal 单点瓶颈 → Canal HA 部署（ZooKeeper 协调）
> 3. ES bulk 写入慢 → 调整 refresh_interval、增加 bulk size
> 4. 对搜索实时性要求不高的场景可以接受 1-3s 延迟

**Q: 为什么用 Canal 而不是直接发 Kafka 消息？**

> 1. **解耦业务代码**：note-service 不需要知道搜索系统的存在，只关心自己的业务逻辑
> 2. **统一数据出口**：MySQL binlog 是唯一真实数据源，所有下游系统（搜索、缓存、数据仓库）都从 Canal 获取，不用分别对接
> 3. **可靠性**：Canal 保证 binlog 解析的位置记录，即使宕机也能从上次位置恢复
> 4. **零侵入**：note-service 只需要写 MySQL（这是它本来就做的），不需要引入 Kafka 生产者代码
>
> 缺点是引入了一个额外的组件（Canal），增加了运维复杂度。但对于已经使用 MySQL 作为源数据的系统，Canal 是成熟且广泛使用的方案。

**Q: Function Score 和纯 BM25 排序的区别？**

> BM25 是纯文本相关性算法：TF（词频）× IDF（逆文档频率）× 文档长度归一化。它只关心文本匹配度。
>
> Function Score 在 BM25 基础上叠加自定义权重：
> - 适合内容平台（搜索"Spring"时，写得好+点赞多的文章排在前面）
> - 关键是要保证文本相关性仍然是主排序因子——如果让点赞数权重太高，会出现"搜索 Java 返回 Python 文章"的问题
> - 通常用 `boost_mode: multiply` 或 `replace`，让业务权重在 BM25 分数基础上做乘法

**Q: Completion Suggester 和 Search-as-you-type 的区别？**

> - **Completion Suggester**：前缀匹配，FST 数据结构，毫秒级，"S" → "Spring Boot"。适合搜索框输入补全。
> - **Search-as-you-type**：实时搜索，每输入一个字符就执行全文搜索。支持中缀/模糊匹配，但延迟高，适合小数据集。
> - **Edge N-gram**：索引时把每个词拆成前 N 个字符，"spring" → ["s", "sp", "spr", "spri", "sprin", "spring"]。查询时用 term filter 匹配。存储开销大，但支持中缀匹配。
> - 本项目选择 Completion Suggester，因为补全场景不需要模糊匹配。

**Q: 如果要在搜索时做个性化排序（如关注用户的笔记优先），怎么改？**

> 1. Function Score 中加 `filter` 函数：匹配关注用户 ID 的文档获得额外 boost
> 2. 或者在 ES 文档中存一个 `following_ids` 字段，用 script_score 判断当前用户是否在其中
> 3. 更复杂的需求用 Learning to Rank（LTR）插件，训练排序模型

**Q: 为什么不把评论也加入 ES 索引？**

> 1. 评论数据存在 Cassandra，Canal 无法直接监控
> 2. 搜索场景主要是"找笔记"和"找人"，评论搜索是次要需求
> 3. 如果后续需要，评论可以走单独的数据通路（Cassandra → Kafka → ES），不经过 Canal
