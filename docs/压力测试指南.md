# 压力测试指南 — 以点赞接口为例

## 1. 什么是压力测试

压力测试（Stress Testing）是**逐步增加并发负载，直到系统达到瓶颈或崩溃**，以确定系统的极限容量和薄弱环节。不同于负载测试（验证预期负载下是否正常），压力测试的目的是**找到断点**。

### 关键指标

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| **QPS / TPS** | 每秒请求数 / 每秒事务数 | 总请求数 ÷ 测试时长 |
| **RT (Response Time)** | 响应时间 | 从发送到收到响应的时间 |
| **P50 / P90 / P99** | 分位数延迟 | 50% / 90% / 99% 的请求在多少 ms 内完成 |
| **Error Rate** | 错误率 | 失败请求数 ÷ 总请求数 × 100% |
| **并发数 (Concurrency)** | 同时进行的请求数 | 模拟的虚拟用户数 |
| **吞吐量** | 单位时间处理的数据量 | QPS × 平均响应大小 |

**最重要的是 P99**，不是平均值。平均值会被极端值拉偏，P99 告诉你"99% 的用户体验到了什么"。

## 2. 点赞接口架构回顾

在进行压力测试之前，先理解被压测系统的完整链路：

```
POST /api/interaction/toggle
  │
  ▼
[Gateway :8080]
  ├── JWT 鉴权（@AuthenticationPrincipal 提取 userId）
  └── @DubboReference → InteractionRpcService.toggle()
        │
        ▼ (Dubbo Triple RPC, Nacos 负载均衡)
[note-service]
  ├── InteractionService.toggle()
  │     ├── targetType = "note" ?
  │     │     └── execute Lua script (Redis, 单线程原子操作)
  │     │           ├── SISMEMBER: 检查用户是否已赞
  │     │           ├── SADD/SREM: 添加/移除集合
  │     │           └── INCR/DECR: 更新计数器
  │     └── targetType = "comment" ?
  │           └── @Async → CommentLikeService.toggleAsync()
  │                 ├── Cassandra comment_like 表 INSERT/UPDATE
  │                 └── Cassandra comment_like_count 表 counter 更新
  ├── InteractionEventProducer.publish() (Kafka 异步, note only)
  │     └── KafkaTemplate.send("interaction-events", ...)
  │           │
  │           ▼ (Kafka Broker :9092)
  │     InteractionEventConsumer (每 30s 窗口聚合)
  │           └── MySQL interaction_record 批量 UPSERT
  └── 返回 ToggleResponse {active, count, action}
```

### 瓶颈预判

阅读架构后，可以预判每个环节的瓶颈在哪里：

| 环节 | 预判瓶颈 | 原因 |
|------|---------|------|
| **Gateway → Dubbo** | 🟡 网络 + 序列化 | Dubbo Triple 协议，gRPC 级性能，通常不是瓶颈 |
| **Redis Lua** | 🟢 极低 | 单线程内存操作，微秒级。单节点轻松 10 万 QPS |
| **Kafka 异步发送** | 🟡 可能堆积 | `@Async` + `whenComplete` 回调，不阻塞主线程。但 Kafka 写入需要网络 I/O |
| **Cassandra** (评论点赞) | 🟡 分区压力 | 如果同一 comment_id 被大量并发点赞，Coordinator 节点压力集中 |
| **JWT 鉴权** | 🟢 低 | 无状态 Token 解析，纯 CPU 计算 |
| **Spring Security Filter Chain** | 🟡 中等 | 每个请求经过过滤器链（认证 → 授权 → 路由） |

**预判结论**：点赞接口最可能的瓶颈是 **Redis 热点 Key**（同一篇笔记的计数器被大量并发更新）和 **Kafka 生产者缓冲区**。

## 3. 压力测试流程

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 1. 计划   │ → │ 2. 准备   │ → │ 3. 预热   │ → │ 4. 执行   │ → │ 5. 分析   │
│          │   │          │   │          │   │          │   │          │
│ 定目标    │   │ 搭环境    │   │ 确认系统  │   │ 逐步加压  │   │ 找瓶颈    │
│ 定场景    │   │ 造数据    │   │ 排除冷启动 │   │ 记录指标  │   │ 出报告    │
│ 定工具    │   │ 写脚本    │   │          │   │ 观察告警  │   │ 定优化    │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

### 3.1 计划阶段 — 定义目标

**点赞接口压测目标**：

| 维度 | 目标值 | 依据 |
|------|--------|------|
| 目标 QPS | 5000 | 假设峰值：10000 在线 × 5% 同时点赞 |
| P99 延迟 | ≤ 50ms | 点赞需要即时反馈，不能有卡顿感 |
| 错误率 | ≤ 0.1% | 几乎不丢请求 |
| 测试时长 | 10 分钟 | 排除瞬时波动，观察稳态表现 |
| 并发梯度 | 100 → 500 → 1000 → 2000 → 5000 | 逐步加压，定位断点 |

**场景定义**：

```
场景 A: 分散点赞（正常流量）
  1000 个不同用户，点赞 1000 篇不同笔记
  → 模拟日常流量，Redis 无热点

场景 B: 热点点赞（最坏情况）
  1000 个不同用户，全部点赞同一篇热门笔记（noteId=42）
  → 模拟爆款笔记，Redis 单 Key 热点
  → 这才是真正的压力测试！
```

### 3.2 准备阶段 — 环境与数据

**环境要求**：

```
压测环境:
  ├── JMeter 客户端      (独立机器，不与被测系统共享资源)
  ├── 被压测系统          (Docker Compose 完整环境)
  ├── InfluxDB + Grafana (监控指标采集)
  └── 网络                (建议有线，避免 WiFi 抖动干扰)
```

**前置数据准备**：


### 3.3 预热阶段

在正式压测前，需要预热系统：

```
1. 确认所有服务健康
   docker compose ps  → 所有服务 healthy

2. 确认 Redis 连接正常
   docker exec redis redis-cli PING  → PONG

3. 确认 Kafka Topic 存在
   docker exec kafka kafka-topics.sh --bootstrap-server kafka:9092 --list
   → interaction-events, search_sync

4. 小流量预热（100 QPS，持续 30s）
   让 JIT 编译热点代码、连接池填满、Page Cache 加载

5. 确认 ES 索引存在
   curl http://localhost:9200/_cat/indices
   → notes, users
```


```
├── Thread Group: 分散点赞场景

│   ├── HTTP Request Defaults (默认值)
│   ├── HTTP Header Manager (Content-Type: application/json)
│   ├── User Defined Variables (noteId pool)
```

```
HTTP Request 配置:

HTTP Header Manager:
  Content-Type:       application/json

```

用多个 Thread Group 或 Ultimate Thread Group 插件实现：

```
阶段 1: 100 并发  → 持续 2 分钟  (基准)
阶段 2: 500 并发  → 持续 2 分钟
阶段 3: 1000 并发 → 持续 2 分钟
阶段 4: 2000 并发 → 持续 2 分钟
阶段 5: 5000 并发 → 持续 2 分钟  (或直到系统崩溃)

每阶段间隔 30s，让系统稳定
```

### 5.2 分散点赞

```bash
# 先获取一个有效 JWT Token
TOKEN=$(curl -s -X POST http://localhost:8080/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"testuser","password":"test123"}' \
  | jq -r '.data.token')

# 准备 Lua 脚本（随机选 noteId）
cat > toggle_random.lua << 'LUA'
noteIds = {}
for i = 1, 10000 do
    noteIds[i] = tostring(1000 + i)
end

request = function()
    local noteId = noteIds[math.random(#noteIds)]
    local body = string.format(
        '{"interactionType":"LIKE","targetType":"note","targetId":%s}',
        noteId
    )
    return wrk.format("POST", "/api/interaction/toggle",
        {["Content-Type"] = "application/json",
         ["Authorization"] = "Bearer " .. token},
        body)
end

-- 每秒打印一次统计
done = function(summary, latency, requests)
    io.write(string.format(
        "QPS: %.0f, P50: %.1fms, P99: %.1fms, Errors: %d\n",
        summary.requests / (summary.duration / 1000000),
        latency:percentile(50) / 1000,
        latency:percentile(99) / 1000,
        summary.errors.status))
end
LUA

# 执行压测
wrk -t8 -c500 -d5m -s toggle_random.lua http://localhost:8080
```

### 6.2 关键监控指标

| 层级 | 指标 | 正常值 | 告警值 |
|------|------|--------|--------|
| **JVM** | `jvm_memory_used_bytes` (heap) | < 80% max | > 90% → 可能 OOM |
| **JVM** | `jvm_gc_pause_seconds` (P99) | < 50ms | > 200ms → GC 压力 |
| **JVM** | `jvm_threads_live_threads` | < 500 | > 1000 → 线程泄漏 |
| **Tomcat** | `tomcat_threads_busy` | < 80% max | = max → 请求排队 |
| **DB** | `hikaricp_connections_active` | < 10 | = max → 连接池耗尽 |
| **Redis** | `lettuce_command_latency_seconds` | < 5ms | > 20ms → Redis 慢 |
| **Kafka** | `spring_kafka_producer_record_send_total` | 等于请求数 | 下降 → 发送失败 |
| **系统** | `system_cpu_usage` | < 70% | > 90% → CPU 瓶颈 |
| **系统** | `jvm_memory_used_bytes` / container limit | < 80% | > 90% → 即将 OOMKilled |

## 7. 执行 — 完整命令序列

```bash
#!/bin/bash
# 文件: scripts/stress-test-like.sh
# 点赞接口压力测试 — 执行脚本

set -e

BASE_URL="http://localhost:8080"
RESULT_DIR="stress-results/$(date +%Y%m%d_%H%M%S)"
mkdir -p "$RESULT_DIR"

echo "=== 1. 环境检查 ==="
curl -s "$BASE_URL/actuator/health" | jq .
docker compose ps | grep -v "Up" && echo "WARNING: 有服务未启动"

echo "=== 2. 数据准备: 获取测试 Token ==="
TOKEN=$(curl -s -X POST "$BASE_URL/api/auth/login" \
  -H 'Content-Type: application/json' \
  -d '{"username":"testuser","password":"test123"}' \
  | jq -r '.data.token')
echo "Token: ${TOKEN:0:20}..."

echo "=== 3. 预热 (100 QPS, 30s) ==="
wrk -t4 -c100 -d30s \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"interactionType":"LIKE","targetType":"note","targetId":1001}' \
  "$BASE_URL/api/interaction/toggle" | tee "$RESULT_DIR/01-warmup.txt"

echo "=== 4. 分散点赞 (500 并发, 5min) ==="
wrk -t8 -c500 -d300s \
  -H "Authorization: Bearer $TOKEN" \
  -s scripts/wrk_random_note.lua \
  "$BASE_URL/api/interaction/toggle" | tee "$RESULT_DIR/02-scattered.txt"

echo "=== 5. 热点点赞: 100 并发 ==="
wrk -t4 -c100 -d120s \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"interactionType":"LIKE","targetType":"note","targetId":42}' \

echo "=== 6. 热点点赞: 500 并发 ==="
wrk -t8 -c500 -d120s \
  -H "Authorization: Bearer $TOKEN" \

echo "=== 7. 热点点赞: 1000 并发 ==="
wrk -t12 -c1000 -d120s \
  -H "Authorization: Bearer $TOKEN" \

  -H "Authorization: Bearer $TOKEN" \
```

### 8.2 瓶颈定位决策树

```
P99 延迟 > 100ms?
  ├── Tomcat 线程数满?  (tomcat_threads_busy = max)
  │     └── 增加 tomcat.threads.max 或增加实例数
  ├── Redis 延迟高?  (lettuce_command_latency > 20ms)
  │     └── 检查 Redis 是否 CPU 100%，考虑 Redis Cluster
  ├── GC 频繁?  (jvm_gc_pause_seconds 频繁 > 100ms)
  │     └── 增大堆内存，或调优 GC 参数（G1GC）
  ├── Dubbo 线程池满?  (dubbo_thread_pool_active = max)
  │     └── 增加 dubbo.provider.threads
  └── Kafka 发送缓冲区满?
        └── 增大 buffer.memory 或 linger.ms

错误率 > 1%?
  ├── Sentinel 限流触发?
  │     └── 提高限流阈值 或 增加节点
  ├── 连接池耗尽?  (hikaricp_connections_active = max)
  │     └── 增大连接池 或 优化慢查询
  └── Dubbo 超时?
        └── 增大 dubbo.provider.timeout
```

### 8.3 预期结果分析（基于架构预判）

**分散点赞**应该表现极好：

```
预判: 高 QPS（3 万+），低延迟（P99 < 30ms）
原因: Redis Lua 原子操作，微秒级完成。不同笔记分布在不同 Key 上，无竞争。
瓶颈: 可能在 Gateway 的 Tomcat 线程数或 Dubbo 连接池。
```

**热点点赞**会暴露问题：

```
预判: QPS 下降（可能 < 5000），P99 升高（可能 > 100ms）
原因: 所有请求争抢同一个 Redis Key（note:like:count:note:42），
      Redis 单线程串行化这些操作，形成排队。
这是设计预期行为 — Redis 单线程保证了原子性，代价是热点串行。

如果在接受范围内 → 不需要优化
如果不可接受 → 考虑：
  1. Redis Cluster 分散热点（按 user_id hash 到不同节点）
  2. 本地缓存 + 异步合并（复杂，一致性弱）
  3. 加机器（Redis 单机已经是微秒级，加机器无帮助 — 热点 Key 仍在同一节点）
```

### 8.4 性能拐点识别

```
将各梯度的 QPS 和 P99 画成折线图:

QPS ↑
    │        ┌─ 系统饱和，QPS 不再增长
    │       ╱
    │      ╱
    │     ╱  ← 拐点：最优并发（QPS 最高，延迟可接受）
    │    ╱
    │   ╱
    │  ╱
    └────────────────────→ 并发数

P99 ↑
    │              ┌─ 急剧恶化
    │             ╱
    │            ╱
    │           ╱
    │       ───╱  ← 拐点：延迟开始非线性增长
    │  ───────╱
    │────────╱
    └────────────────────→ 并发数

结论: 拐点之前 = 安全容量，拐点之后 = 过载
```

## 10. 常见问题

### Q: 压测结果中 QPS 很高但实际生产很低？

可能原因：
- 压测网络延迟接近于 0（同机或同机房），生产走公网
- 压测用的 JWT Token 都一样 → Gateway 的鉴权缓存命中率 100%
- 生产有更多中间件（API Gateway、SLB、CDN）

**对策**：压测环境尽量贴近生产，用不同 Token 模拟不同用户。

### Q: 为什么 Redis 热点 Key 是瓶颈但 CPU 没满？

Redis 是单线程处理命令。热点 Key 导致所有请求排队在一个 CPU 核心上，其他核心空闲。这是 Redis 的设计特性，不是 Bug。

### Q: 压力测试多久跑一次？

- 每次大版本发布前：全量压测
- 每次架构变更（加缓存、换中间件）：针对性压测
- 每周：核心接口快速压测（单场景，5 分钟）
- 不跑：每个 PR（太重了）

### Q: wrk 和 JMeter 怎么选？

| | wrk | JMeter |
|---|---|---|
| **安装** | 编译或单文件 | 解压即用，依赖 Java |
| **上手** | 5 分钟 | 1 小时 |
| **脚本能力** | Lua 脚本 | GUI + BeanShell/JSR223 |
| **报告** | 终端输出 | HTML Dashboard + 图形 |
| **分布式** | 不支持 | 支持（多台机器同时压） |
| **适合** | 快速验证、API 压测 | 完整测试计划、复杂场景 |

> 建议：日常用 wrk 快速验证，发版前用 JMeter 出完整报告。
