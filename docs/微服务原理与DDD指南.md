# 微服务原理与 DDD 指南

## 1. 单体 vs 微服务

### 1.1 单体架构

```
┌─────────────────────────────────────────┐
│              单体应用 (WAR/JAR)           │
│                                         │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │  用户模块 │ │  订单模块 │ │  支付模块 │   │
│  └─────────┘ └─────────┘ └─────────┘   │
│         ↓         ↓         ↓          │
│        ┌───────────────────┐           │
│        │    同一个数据库      │           │
│        └───────────────────┘           │
└─────────────────────────────────────────┘
```

| 优点 | 缺点 |
|------|------|
| 开发简单，一个 IDE 搞定 | 代码膨胀，编译越来越慢 |
| 部署简单，一个 war 包 | 牵一发动全身，改一行部署全部 |
| 测试简单，一次启动 | 无法独立扩缩容（秒杀和后台挤在一起） |
| 适合早期小团队快速迭代 | 技术栈锁定，不能局部引入新技术 |

### 1.2 微服务架构

```
┌─────────┐  ┌─────────┐  ┌─────────┐
│ 用户服务  │  │ 订单服务  │  │ 支付服务  │
│ 独立部署  │  │ 独立部署  │  │ 独立部署  │
│ 独立数据库 │  │ 独立数据库 │  │ 独立数据库 │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │
     └─────────────┼─────────────┘
                   │
           ┌───────┴──────┐
           │   服务注册/网关  │
           └──────────────┘
```

**核心原则**：

| 原则 | 说明 |
|------|------|
| **独立部署** | 每个服务可独立构建、测试、发布，不影响其他服务 |
| **独立数据源** | 每个服务有自己的数据库，不共享表 |
| **围绕业务能力** | 按业务域拆分，而非技术层拆分 |
| **轻量级通信** | HTTP REST / RPC / 消息队列 |
| **去中心化治理** | 每个服务选最适合的技术栈 |

## 2. 微服务的核心问题与解决模式

拆成微服务后，单体中不存在的问题全部浮现：

### 2.1 服务发现

**问题**：服务 A 怎么知道服务 B 在哪里？

```
单体: 方法调用，编译时确定
微服务: B 的 IP 和端口在变（扩缩容、重启），不能硬编码
```

**解决（Nacos / Eureka）**：

```
服务 B 启动
  → 向 Nacos 注册: "upload-service → 192.168.1.5:20880"
服务 A 调用 B
  → 从 Nacos 查询: "upload-service 有哪些实例？"
  → Dubbo 负载均衡选一个
  → 调用 192.168.1.5:20880
B 下线
  → Nacos 心跳检测失败
  → 剔除注册信息
  → A 下次查询拿不到已下线实例
```

**本项目**：Nacos 做注册中心，Dubbo 做 RPC 框架。

### 2.2 配置管理

**问题**：10 个服务的配置散落各处，改一个数据库地址要改 10 遍？

**解决（Nacos Config）**：所有配置收敛到 Nacos，服务启动拉取，配置变更实时推送。

### 2.3 服务间通信

| 方式 | 协议 | 适合场景 | 本项目使用 |
|------|------|---------|-----------|
| **RPC** | Dubbo Triple | 同步调用，实时响应 | ✅ gateway → 业务服务 |
| **HTTP REST** | HTTP/JSON | 对外开放、跨语言 | ✅ 前端 → gateway |
| **消息队列** | Kafka | 异步、削峰、解耦 | ✅ Canal → ES 同步 |

### 2.4 分布式事务

**问题**：一个操作跨两个服务，如何保证一致性？

```
单体: @Transactional → BEGIN; UPDATE A; UPDATE B; COMMIT;
微服务: 服务A写入成功 + 服务B写入失败 → 数据不一致
```

**解决模式**：

| 模式 | 说明 | 本项目 |
|------|------|--------|
| **Seata AT** | 自动回滚，侵入小 | — |
| **Saga** | 正向操作 + 补偿操作链 | — |
| **最终一致性** | 放弃强一致，通过异步 + 重试保证最终一致 | ✅ Canal → Kafka → ES |

> 能不用分布式事务就不用。绝大多数场景通过设计规避——数据归一个服务管。

### 2.5 网关

**问题**：前端要调 10 个服务的不同端口？鉴权逻辑写 10 遍？

**解决（Spring Cloud Gateway）**：统一入口，统一鉴权、限流、路由。

```
前端 → gateway:8080/api/note/detail
  → Auth Guard (JWT 校验)
  → Sentinel (限流)
  → Dubbo Reference → note-service
```

### 2.6 容错与限流

**问题**：服务 B 挂了，服务 A 等着超时 → 线程池耗尽 → 服务 A 也挂了 → 雪崩。

**解决（Sentinel）**：熔断（快速失败）、降级（返回兜底）、限流（拒绝超额请求）。

## 3. 如何拆分微服务

拆分是微服务最难的问题，搞错了比单体还糟糕。

### 3.1 错误拆分方式

| 方式 | 为什么错 |
|------|---------|
| **按技术层拆分** | `controller-service` + `dao-service` → 每次改动跨 3 个服务 |
| **按表拆分** | 一个表一个服务 → 所有请求都要跨服务 JOIN |
| **拆太细** | 100 个服务各 50 行代码 → 运维噩梦 |
| **拆太粗** | "核心服务"包含 80% 逻辑 → 换了名字的单体 |

### 3.2 正确的拆分依据：业务边界

```
不是拆"订单表服务" + "用户表服务"
而是拆"交易域"     + "用户域"
      ↓                ↓
   订单、支付、退款    注册、登录、个人资料
   这是一个业务领域    这是另一个业务领域
```

**判断标准**：

| 信号 | 说明 |
|------|------|
| **独立业务语言** | 交易域说"下单/支付/退款"，用户域说"注册/登录/实名" |
| **独立变更节奏** | 用户模块一周改一次，支付模块一年改一次 |
| **独立团队** | 两个团队各自负责不同业务线 |
| **独立数据所有权** | 订单表只归订单服务，其他服务不能直连 |

### 3.3 本项目的拆分

| 服务 | 业务域 | 独立数据库 |
|------|--------|-----------|
| `auth-service` | 用户身份认证 | ✅ MySQL（user 相关表） |
| `note-service` | 笔记内容 + 评论 | ✅ Cassandra + Redis |
| `upload-service` | 文件上传 | ✅ MinIO（对象存储） |
| `leaf-service` | 分布式 ID 生成 | ✅ MySQL（leaf_alloc 表） |
| `search-service` | 搜索查询 | —（读 ES） |
| `search-sync-service` | 数据索引管道 | —（写 ES） |
| `gateway` | API 网关 | —（无状态路由） |

## 4. DDD（领域驱动设计）

### 4.1 为什么需要 DDD？

**微服务告诉你想"拆"，DDD 告诉你想"怎么拆"**。

一个典型问题：团队决定拆微服务，第一个反应是"数据库有哪些表？我们按表拆。"然后发现所有表都互相依赖，拆不动了。

DDD 的核心洞察：**先理解业务，再设计代码**。代码的边界应该和业务的边界一致。

### 4.2 核心概念

#### 通用语言（Ubiquitous Language）

团队使用同一套词汇，代码命名和业务讨论一致。

```
❌ 混乱: 业务说"下单"，数据库叫 order_create，代码叫 submitTransaction
✅ 统一: 全部叫 PlaceOrder
```

#### 限界上下文（Bounded Context）— 最重要的概念

**同一个词在不同上下文中有不同含义。**

```
"用户"
  在认证上下文中: 用户名 + 密码 + 手机号
  在笔记上下文中: 用户 ID + 昵称 + 头像
  在支付上下文中: 账户余额 + 实名信息

三个上下文各有自己的 "User" 模型，共享 ID 但不共享字段。
```

**限界上下文 = 微服务的自然边界**：

```
限界上下文 → 微服务
认证上下文 → auth-service
笔记上下文 → note-service
```

#### 实体（Entity） vs 值对象（Value Object）

| | Entity | Value Object |
|---|---|---|
| **标识** | 有唯一 ID | 无 ID，值本身决定相等性 |
| **可变性** | 可变 | 不可变 |
| **示例** | User、Order | Money、Address、Email |
| **持久化** | 自己一张表 | 嵌入所属实体 |

#### 聚合（Aggregate）与聚合根（Aggregate Root）

**聚合**：一组紧密相关的对象，作为一个整体操作。

**聚合根**：聚合的入口，外部只能通过聚合根访问聚合内部对象。

```
聚合: Note (聚合根)
  ├── NoteContent (值对象)
  ├── Comment[] (实体集合)
  └── NoteStats (值对象: viewCount, likeCount)

规则:
  ✅ CommentRepository.findByNoteId(noteId)  — 通过聚合根查
  ❌ CommentService.publishComment(comment)  — 直接绕过聚合根
  → 应该: Note.publishComment(comment) → CommentRepository.save()
```

**一句话**：聚合根保证业务规则不被破坏。点赞不能点给一篇删除的笔记——如果直接操作 Comment 绕过了 Note，这个规则就丢了。

#### 领域事件（Domain Event）

聚合内发生的重要业务事实，通知其他上下文。

```
Note.Published → 发布领域事件
  → Canal 订阅 binlog（"异曲同工"的领域事件实现）
  → ES 更新索引
  → （未来）推荐系统感知到新内容
```

#### 仓储（Repository）

给聚合根提供持久化能力，隐藏数据库细节。

```java
// 接口定义在领域层（不依赖任何框架）
public interface NoteRepository {
    Optional<Note> findById(Long noteId);
    void save(Note note);
    void delete(Note note);
}

// 实现在基础设施层（Spring Data Cassandra）
@Repository
class NoteCassandraRepository implements NoteRepository { ... }
```

#### 分层架构

```
┌──────────────────────────────┐
│       用户接口层 (API)        │  Controllers, DTOs
├──────────────────────────────┤
│       应用层 (Application)    │  编排领域对象，事务管理
├──────────────────────────────┤
│       领域层 (Domain)         │  Entity, ValueObject, Aggregate,
│                              │  DomainService, 领域规则
├──────────────────────────────┤
│       基础设施层 (Infra)      │  Repository 实现, 数据库, MQ, RPC
└──────────────────────────────┘

依赖方向: 上层依赖下层，领域层不依赖任何外部框架
```

**核心原则**：领域层是纯 Java 代码，不 import Spring、Dubbo、JDBC——这样领域逻辑可以独立单元测试，不被基础设施绑架。

## 5. DDD 与本项目的对照

| DDD 概念 | 本项目对应 |
|----------|-----------|
| **限界上下文** | `auth-service`（认证上下文）、`note-service`（笔记上下文） |
| **聚合根** | Note（笔记聚合，管理评论和统计） |
| **值对象** | `NoteStats {viewCount, likeCount}` |
| **领域事件** | MySQL binlog → Canal → Kafka → ES Indexing |
| **仓储** | `NoteRepository.java`（接口） + `NoteCassandraRepository`（实现） |
| **应用服务** | `NoteService.java`（编排 draft/publish/detail） |
| **通用语言** | `publishNote`、`createDraft`、`toggleLike` |

### 观察：Canal binlog 是天然的事件溯源

DDD 推崇领域事件（Aggregate 状态变更即产生事件）。本项目 MySQL + Canal 的组合恰好实现了这个模式：

```
note-service publishNote()
  → Note 聚合修改状态（draft → published）
  → 持久化到 MySQL
  → MySQL binlog = "事件存储"
  → Canal 解析 binlog = "事件发布"
  → Kafka = "事件总线"
  → search-sync-service = "事件消费者"
```

业务代码不需要显式发布事件——MySQL binlog 自动捕获所有变更。

## 6. 微服务与 DDD 的关系总结

```
DDD 回答 WHAT:
  业务的边界在哪里？
  哪些概念属于一起？
  规则由谁负责？

微服务回答 HOW:
  怎么部署这些边界？
  服务间怎么通信？
  怎么保证可靠性？

实践路径:
  1. DDD 划分限界上下文 → 得到微服务拆分方案
  2. DDD 识别聚合根     → 得到数据库所有权边界
  3. DDD 定义领域事件   → 得到服务间异步协作方案
  4. 微服务基础设施     → 注册、配置、网关、容错落地
```

## 7. 常见问题

### Q: 微服务一定要用 DDD 吗？

不一定。CRUD 为主的系统用传统三层架构（Controller-Service-DAO）够用。DDD 适合**业务规则复杂的系统**——支付风控、保险理赔、供应链管理。

### Q: 多大算"微"？

没标准答案。亚马逊的标准是"一个团队能在两周内重写这个服务"。本项目的服务大概 300-1000 行核心逻辑。

### Q: 服务间能不通信尽量不通？

对。原则：**"先想能不能各干各的，再想怎么通信。"** 本项目设计上 note-service 不知道 ES 的存在——这个解耦是 Canal 带来的。

### Q: 分布式事务到底怎么处理？

三种思路：
1. **规避设计**：让操作归一个服务、一个数据库
2. **最终一致性**：异步消息 + 重试 + 幂等（本项目的选择）
3. **Seata/Saga**：真正的分布式事务框架（复杂度高，非必要不引入）

### Q: 什么时候不该拆微服务？

- 团队 < 5 人 → 单体更快
- 产品还在探索 PMF → 不需要微服务的复杂度
- 数据强关联（所有表 JOIN 所有表）→ 拆了更糟
